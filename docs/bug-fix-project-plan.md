## 4. 协作规范与模板（建议在 `0.2` 中确认）

### 4.1 标准 Bug 描述模板（必备字段）
每条 Bug 必须结构化，避免研发无法复现或无法判断优先级：

```
【标题】一句话描述现象
【复现环境】OS / 浏览器 / 机型 / 网络
【账号信息】测试账号 / 角色 / 权限
【版本】提测版本号 / 构建号
【前置条件】进入该场景前的状态
【复现步骤】1. 2. 3. （可复现的操作序列）
【实际结果】真实发生的行为
【期望结果】应当发生的行为
【截图/录屏】附件
【日志/报错】关键日志、堆栈、接口返回
【影响范围】受影响功能、用户量、是否线上
【严重程度】S1-S4
【优先级】P0-P3
【是否阻断上线】是 / 否
```

### 4.2 严重程度（Severity）定义
| 级别 | 定义 |
| --- | --- |
| **S1 致命** | 线上核心功能不可用 / 数据丢失 / 资金或安全事故 |
| **S2 严重** | 主要功能缺陷，无合理绕过方案 |
| **S3 一般** | 功能缺陷但有绕过方案，或次要功能异常 |
| **S4 轻微** | UI/文案/体验类，不影响主流程 |

### 4.3 优先级（Priority）与阻断判定
| 级别 | 含义 |
| --- | --- |
| **P0** | 必须立即处理，阻断上线 |
| **P1** | 本版本内必须修复 |
| **P2** | 本版本尽量修复，可评估后延期 |
| **P3** | 可排期至后续版本 |

> **阻断上线规则**：S1 全部阻断；S2 默认阻断，除非有产品书面豁免；涉及线上事故时，优先确认影响范围、止损方案与回滚预案。

### 4.4 测试用例模板（六类场景）
| 用例ID | 模块 | 场景类型 | 前置条件 | 步骤 | 期望 | 优先级 | 负责人 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| TC-xxx |  | 正常/边界/异常/权限/兼容/回归 |  |  |  |  |  |

### 4.5 状态流转约定（建议）
`待办 → 进行中 → 待验证 → 已解决/已关闭`；研发定位与修复在「进行中」；测试在「待验证」认领回归；验收通过转「已关闭」。

## 5. 阶段流转与里程碑

```
阶段一(测试准备) ──▶ 阶段二(Bug收集分级) ──▶ 阶段三(研发定位)
        │                  │                      │
        ▼                  ▼                      ▼
   测试用例/计划       Bug清单+分级            复现+可能根因
                                              │
             阶段四(修复验证) ◀── 研发修复 ────┘
                    │
                    ▼
             阶段五(测试验收) ──▶ 验收结论/质量报告
                    │
                    ▼
             阶段六(质量复盘) ──▶ 改进项/规范沉淀
```

**进入下一阶段的前置门槛**
- 阶段一→二：测试计划与用例已评审，环境就绪。
- 阶段二→三：Bug 已结构化且完成分级，研发已认领。
- 阶段三→四：研发提交修复说明与影响范围。
- 阶段四→五：回归全部通过，遗留问题已记录。
- 阶段五→六：验收结论产出，遗留问题转为后续事项。
