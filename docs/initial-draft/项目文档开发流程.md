# FiveAvenues 项目文档开发流程

> 适用范围：FiveAvenues（五大道旅行文化指南）仓库及关联团队知识库
> 制定依据：基于本仓库既有文档实践（`REQUIREMENTS.md` / `DESIGN.md` / `docs/` / `.workbuddy/memory/` / `rag Design/` 等）归纳而成，非凭空设计
> 文档状态：⚠️ 第一稿，建议由 PO 与技术负责人评审后生效

---

## 1. 目的与原则

- **目的**：让项目文档成为「可追溯、可评审、可过期治理」的活体系，而非一次性上传物。每份文档都能回答：讲什么、给谁看、现在是否有效、谁负责更新。
- **核心原则**：
  1. **需求可追溯**——每条功能/设计改动都能回溯到 Issue / PR / 决策记录。
  2. **单一事实源**——主需求在 `REQUIREMENTS.md`，版本范围在 `docs/PRD_Vx.y_*`，不允许多处各写一份"真相"。
  3. **先文档后实现**——新版本先有 PRD 与研发拆解，再进入编码。
  4. **实现即同步文档**——代码合并时同步更新 `DESIGN.md` / `REQUIREMENTS.md` 状态与变更日志。
  5. **过期必标注**——探索性/过时文档显式归档，不静默遗留（参考 `rag Design/00-已过时归档说明.md`）。

## 3. 文档生命周期（五阶段）

```
① 规划/立项 ──► ② 撰写/评审 ──► ③ 实现同步 ──► ④ 验证/交付 ──► ⑤ 归档/过期治理
   PRD 起草        创建评审事项      代码合并同步      冒烟+交付总结      状态标注+定期复查
   关联 Issue      添加关注人        更新状态/日志     更新变更日志        过期文档标"已废弃"
```

| 阶段 | 动作 | 产出 | 负责人 |
| --- | --- | --- | --- |
| ① 规划 | 明确范围、写 PRD（EARS）、列待确认项 A1~An | `docs/PRD_Vx.y_*.md` | 产品需求助手/PO |
| ② 评审 | 创建评审事项，分配技术/设计/测试，附 PRD 为交付物 | 评审事项 + 结论 | PO |
| ③ 实现 | 编码 + 同步 `DESIGN.md`/`REQUIREMENTS.md` 状态与变更日志 | 代码 + 文档更新 | 研发 |
| ④ 验证 | 端到端冒烟 + 写交付总结 + 回填变更日志 | `overview-frN.md`、测试清单 | 研发/测试 |
| ⑤ 治理 | 标注状态（有效/待更新/已废弃/仅供参考），定期复查 | 状态标记、归档说明 | 知识库维护人 |

## 4. 写作规范

### 4.1 需求写法（EARS）
采用本仓库既有的 EARS 五类写法，需显式标注类型：
- **Ubiquitous（普遍约束）**：系统**应始终**……
- **Event-driven（事件触发）**：当……时，系统**应**……
- **State-driven（状态驱动）**：在……态下，系统**应**……
- **Optional（可选）**：Where……，系统**应**……
- **Unwanted（异常）**：若……，则系统**应**……（不静默失败）

每条需求含：编号、优先级（P0/P1/P2）、用户故事、约束、状态、关联 Issue/PR。

### 4.2 状态标记约定
| 标记 | 含义 |
| --- | --- |
| ✅ 已实现 | 代码完成且通过验证 |
| 🔲 规划中 | PRD 阶段，未实现 |
| ⚠️ 降级 | 功能可用但部分降级（如 RAG 不可用回退规则） |
| 已废弃 / 仅供参考 | 过时文档，见归档说明 |

### 4.3 编号与追溯
- 需求：`FR-` / `NFR-`；接口：`/api/...`；测试：`TC-` / `REG-`；决策：`3216ed6` 等 commit。
- 文档头部统一带「**关联文档 / 来源**」区块，确保可追溯（参考 `docs/PRD_V1.2` 头部）。

### 4.4 版本号
- 主需求台账随实现更新状态；版本级 PRD/拆解以 `Vx.y` 命名，如 `PRD_V1.2_RAG社区增强.md`、`V1.2_研发任务拆解.md`。
- 单需求交付总结以功能编号命名：`overview-fr8.md`。

## 6. 过期治理（重点）

- **触发条件**：业务/流程变化、依赖变更、实现与文档不符、超过约定复核周期。
- **动作**：
  1. 仍可读的旧稿 → 顶部加状态标记「⚠️ 仅供参考 / 已废弃」，并写明被哪份文档取代。
  2. 探索性目录（如 `rag Design/`）→ 新增 `00-已过时归档说明.md` 说明归档原因与时间。
  3. 冲突内容 → 以 `DESIGN.md`/`REQUIREMENTS.md` 单一事实源为准，旧文降级。
- **复查节奏**：建议每版本发布后 + 每季度，由知识库维护人出「过期文档清单」。

## 7. 维护责任与节奏

| 角色 | 职责 |
| --- | --- |
| PO / 产品需求助手 | PRD 撰写与评审组织、待确认项（A1~An）推进 |
| 技术负责人 | 设计裁决（D1~Dn）、`.env`/RAG/埋点端点确认 |
| 研发接手 | 实现 + 同步 `DESIGN.md`/`REQUIREMENTS.md` + 交付总结 |
| 测试 | 用例/回归清单维护、冒烟记录 |
| 知识库维护人 | 分类/命名合规、过期治理、季度盘点 |

> 建议：将「文档补齐 / 过期治理 / 知识更新」动作**创建为事项**，分配维护人并设置关注人（PO、技术负责人等）。

## 8. 文档开发检查清单（Checklist）

- [ ] 新版本是否先有 `docs/PRD_Vx.y_*.md` 与 `docs/Vx.y_研发任务拆解.md`？
- [ ] PRD 是否采用 EARS、含成功指标与验收标准、列出待确认项 A1~An？
- [ ] 是否创建评审事项并分配技术/设计/测试、添加关注人？
- [ ] 代码合并时是否同步更新 `DESIGN.md` / `REQUIREMENTS.md` 状态与变更日志？
- [ ] 是否产出交付总结（`overview-frN.md`）并回填冒烟结果？
- [ ] 探索性文档是否标注状态或加归档说明？
- [ ] 文档头部是否带「最后更新 / 状态 / 关联来源」？
- [ ] 过期文档是否标注「有效/待更新/已废弃/仅供参考」？
