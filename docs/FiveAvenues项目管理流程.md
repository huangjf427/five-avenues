## 二、项目管理总览

本项目采用 **"文档驱动、需求可追溯、版本化管理"** 的项目管理模式。核心特征：

1. **先文档后实现**：每个版本先有 PRD（产品需求文档）与研发任务拆解，再进入编码阶段
2. **单一事实源**：主需求以 `REQUIREMENTS.md` 为唯一台账，版本范围由 `docs/PRD_Vx.y_*.md` 承载
3. **实现即同步文档**：代码合并时同步更新 `DESIGN.md` / `REQUIREMENTS.md` 的状态与变更日志
4. **过期必标注**：探索性/过时文档显式归档，不静默遗留
5. **EARS 需求写法**：需求采用 Ubiquitous / Event-driven / State-driven / Optional / Unwanted 五类规范表述

## 三、项目管理完整流程（五阶段模型）

```
① 规划/立项 ──► ② 撰写/评审 ──► ③ 实现同步 ──► ④ 验证/交付 ──► ⑤ 归档/过期治理
   PRD 起草        创建评审事项      代码合并同步      冒烟+交付总结      状态标注+定期复查
   关联 Issue      添加关注人        更新状态/日志     更新变更日志        过期文档标"已废弃"
```

### 阶段一：规划/立项

| 步骤 | 动作 | 产出 | 负责人 |
|------|------|------|--------|
| 1.1 明确范围 | 确定版本范围、业务目标、成功指标 | 版本范围定义 | PO / 产品需求助手 |
| 1.2 撰写 PRD | 按 EARS 规范写功能需求，含用户故事、验收标准、待确认项 | `docs/PRD_Vx.y_*.md` | 产品需求助手 |
| 1.3 关联 Issue | 在 GitHub 创建对应 Issue，关联 PRD | GitHub Issue | PO |
| 1.4 列出假设 | 对未确认项标注 A1~An，明确待拍板事项 | PRD §9 范围假设 | 产品需求助手 |

**实操示例**：V1.2 的 PRD（`docs/PRD_V1.2_RAG社区增强.md`）包含：
- 5 个用户故事（US-1~US-5）
- 5 个功能需求（FR-9~FR-13，含 EARS 规则、优先级、权限约束）
- 9 项成功指标（含北极星指标「周行程生成成功用户数」）
- 6 项待确认假设（A1~A6：埋点端点、范围裁剪、敏感词词库、排序权重、存储结构、埋点合规）
- 8 项可判定验收标准

### 阶段二：撰写/评审

| 步骤 | 动作 | 产出 | 负责人 |
|------|------|------|--------|
| 2.1 研发拆解 | 将 PRD 每条 FR 拆为可独立交付/可并行的前后端子任务 | `docs/Vx.y_研发任务拆解.md` | 研发负责人 |
| 2.2 创建评审事项 | 在项目事项系统中创建评审事项，附 PRD 为交付物 | 评审事项 + 关注人 | PO |
| 2.3 分派评审 | 添加技术负责人、设计师、测试为关注人 | 评审通知 | PO |
| 2.4 评审结论 | 对 PRD 范围、待确认项、任务拆解做决定 | 评审结论记录 | 全体评审人 |

**实操示例**：V1.2 研发拆解（`docs/V1.2_研发任务拆解.md`）：
- 将 5 个 FR + 埋点拆为 **19 个子任务**（T-9.1~T-13.4、T-M.1~T-M.4）
- 标注每个任务的前置依赖、验收标准
- 绘制任务依赖总览图
- 列出 7 项风险清单及缓解措施
- 排期建议：P1（FR-9/FR-11/FR-12）先并行，P2（FR-10/FR-13）后启动

### 阶段三：实现同步

| 步骤 | 动作 | 产出 | 负责人 |
|------|------|------|--------|
| 3.1 创建开发分支 | 从 main 创建 `dev/vx.y` 分支 | 开发分支 | 研发 |
| 3.2 编码实现 | 按任务拆解逐条实现，遵循 NFR-1（零依赖） | 代码 | 研发 |
| 3.3 同步设计文档 | 代码合并时更新 `DESIGN.md` 路由/数据模型/模块职责 | `DESIGN.md` 更新 | 研发 |
| 3.4 同步需求文档 | 更新 `REQUIREMENTS.md` 状态（✅ 已实现）与变更日志 | `REQUIREMENTS.md` 更新 | 研发 |
| 3.5 提交 PR | 提交 Pull Request，关联 Issue | GitHub PR | 研发 |

**实操示例**：V1.2 实现（分支 `dev/v1.2`）：
- 新增 `app/src/analytics.js`（埋点）、`app/src/reports.js`（举报工单）
- `reviews.js` 扩展 `helpfulCount/helpfulBy/flagged`、敏感词预检、有用投票、批量审核
- `server.js` 新增 7 个 API 端点
- 前端新增 RAG 状态置灰指引、溯源徽章、有用/举报按钮、审核台举报处理 Tab
- 通过 PR #5、#6、#7、#8 逐步合并至 main

### 阶段四：验证/交付

| 步骤 | 动作 | 产出 | 负责人 |
|------|------|------|--------|
| 4.1 端到端冒烟 | 按测试清单逐条验证功能 | 冒烟结果 | 测试/研发 |
| 4.2 回归测试 | 对已有功能做回归，确认无破坏 | 回归清单 | 测试 |
| 4.3 写交付总结 | 含设计裁决、改动清单、冒烟结果、未覆盖项 | `overview-frN.md` | 研发 |
| 4.4 回填变更日志 | 在 `REQUIREMENTS.md` 变更日志中记录日期/变更/驱动者 | 变更日志条目 | 研发 |

**实操示例**：FR-8 交付总结（`overview-fr8.md`）：
- 4 项关键设计裁决（D1~D4）
- 12 个文件改动清单
- 4 个冒烟场景全部通过
- 2 项未覆盖/后续事项（远程 HTTP 拉取、真实 RAG 端到端验证）

### 阶段五：归档/过期治理

| 步骤 | 动作 | 产出 | 负责人 |
|------|------|------|--------|
| 5.1 状态标注 | 对文档标注：有效/待更新/已废弃/仅供参考 | 状态标记 | 知识库维护人 |
| 5.2 过期归档 | 探索性目录新增 `00-已过时归档说明.md` | 归档说明 | 知识库维护人 |
| 5.3 定期复查 | 每版本发布后 + 每季度出「过期文档清单」 | 复查清单 | 知识库维护人 |
| 5.4 冲突处理 | 以 `DESIGN.md` / `REQUIREMENTS.md` 为单一事实源 | 文档一致性 | 知识库维护人 |

**实操示例**：`rag Design/` 目录：
- 该目录为 RAG 探索性设计文档
- 核心结论落地到 `DESIGN.md` §9 后，原稿新增 `00-已过时归档说明.md`
- 归档说明中写明："本目录为早期 RAG 探索设计，最终结论已沉淀至 `DESIGN.md` §9，本目录仅供参考"

### 4.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 主需求 | `REQUIREMENTS.md`（唯一） | — |
| 设计 | `DESIGN.md`（唯一） | — |

### 4.3 需求写法规范（EARS）

所有功能需求必须采用 EARS 五类写法，显式标注类型：

| 类型 | 模板 | 含义 |
|------|------|------|
| **Ubiquitous（普遍约束）** | 系统**应始终**…… | 无条件持续满足的约束 |
| **Event-driven（事件触发）** | 当……时，系统**应**…… | 特定事件触发的行为 |
| **State-driven（状态驱动）** | 在……态下，系统**应**…… | 特定状态下持续满足的行为 |
| **Optional（可选）** | Where……，系统**应**…… | 可选特性开启时的行为 |
| **Unwanted（异常）** | 若……，则系统**应**…… | 异常/错误场景的处置（不静默失败） |

每条需求需含：编号（FR-/NFR-）、优先级（P0/P1/P2）、用户故事、约束、状态、关联 Issue/PR。

### 4.4 状态标记约定

| 标记 | 含义 |
|------|------|
| ✅ 已实现 | 代码完成且通过验证 |
| 🔲 规划中 | PRD 阶段，未实现 |
| ⚠️ 降级 | 功能可用但部分降级（如 RAG 不可用回退规则） |
| 已废弃 / 仅供参考 | 过时文档，见归档说明 |

### 4.5 文档头部规范

所有文档头部统一包含：

```markdown
# 文档标题
> 最后更新：YYYY-MM-DD
> 版本：Vx.y
> 状态：✅ 已实现 / 🔲 规划中 / ⚠️ 降级 / 已废弃
> 关联文档：……
```

## 六、版本管理流程

### 6.1 版本号规则

- 主需求台账（`REQUIREMENTS.md`）随实现更新状态
- 版本级 PRD/拆解以 `Vx.y` 命名，如 `PRD_V1.2_RAG社区增强.md`
- 单需求交付总结以功能编号命名：`overview-fr8.md`
- 当前项目已交付版本：V1.0 → V1.1.0（FR-8）→ V1.2（FR-9~FR-13）

### 6.2 版本发布流程

```
PRD 撰写 → 评审确认 → 研发拆解 → 分支开发 → PR 合并 → 冒烟验证 → 交付总结 → 版本发布
```

### 6.3 分支策略

- `main`：稳定发布分支
- `dev/vx.y`：版本开发分支（如 `dev/v1.2`）
- 通过 Pull Request 合并至 main

## 八、风险管理

### 8.1 风险等级定义

| 等级 | 说明 | 处理原则 |
|------|------|----------|
| 🔴 高 | 阻塞核心功能或交付 | 优先确认，纳入当前迭代 |
| 🟡 中 | 影响部分功能口径或排期 | 先占位方案，后续迭代完善 |
| 🟢 低 | 影响有限，可延后 | 记录跟踪，定期复查 |

### 8.2 当前已知风险（V1.2 交付后）

| 风险 | 等级 | 说明 | 缓解 |
|------|------|------|------|
| 远程知识库拉取未实现 | 🟡 中 | FR-8 仅支持本地目录 / `file://` URL | 后续版本迭代 |
| 内存会话 | 🟡 中 | 多实例需 Redis/DB，重启后需重新登录 | 生产部署前置反向代理，后续迁移持久化 |
| 埋点端点待定 | 🟡 中 | `ANALYTICS_ENDPOINT` 上报目标待确认 | 技术负责人确认后接入 |
| RAG 端到端未验证 | 🟡 中 | 有效密钥 + 真实 embedding 链路未在生产验证 | 生产环境部署后优先验证 |

## 九、文档治理（过期管理）

### 9.1 过期触发条件

- 业务/流程变化
- 依赖变更
- 实现与文档不符
- 超过约定复核周期（每版本发布后 + 每季度）

### 9.2 过期处理动作

1. 仍可读的旧稿 → 顶部加状态标记「⚠️ 仅供参考 / 已废弃」，写明被哪份文档取代
2. 探索性目录（如 `rag Design/`）→ 新增 `00-已过时归档说明.md` 说明归档原因与时间
3. 冲突内容 → 以 `DESIGN.md` / `REQUIREMENTS.md` 单一事实源为准，旧文降级

### 9.3 分类铁律

> `rag Design/` 这类探索性目录**不得**当作权威设计引用；一旦结论落地进 `DESIGN.md`，原稿须加 `00-已过时归档说明.md` 并置顶说明。

## 十、开发检查清单（Checklist）

在每次版本开发中应逐项检查：

- [ ] 新版本是否先有 `docs/PRD_Vx.y_*.md` 与 `docs/Vx.y_研发任务拆解.md`？
- [ ] PRD 是否采用 EARS、含成功指标与验收标准、列出待确认项 A1~An？
- [ ] 是否创建评审事项并分配技术/设计/测试、添加关注人？
- [ ] 代码合并时是否同步更新 `DESIGN.md` / `REQUIREMENTS.md` 状态与变更日志？
- [ ] 是否产出交付总结（`overview-frN.md`）并回填冒烟结果？
- [ ] 探索性文档是否标注状态或加归档说明？
- [ ] 文档头部是否带「最后更新 / 状态 / 关联来源」？
- [ ] 过期文档是否标注「有效/待更新/已废弃/仅供参考」？
- [ ] 新功能是否符合 NFR-1（零 npm 依赖）约束？
