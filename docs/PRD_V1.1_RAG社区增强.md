# FiveAvenues V1.1 PRD — RAG 增强 + 社区运营增强

> 版本：V1.1（规划中，待评审）
> 阶段：阶段二·PRD 撰写
> 主线：RAG 增强 + 社区运营增强（知识库交付见 FR-8 / Issue #1，不在本 PRD 范围）
> 起草：产品需求全流程协同助手
> 关联文档：`REQUIREMENTS.md`（FR-1~FR-8 / NFR-1~NFR-7）、`DESIGN.md`、`需求分析报告_接手基线.md`
> EARS 约定：Ubiquitous（普遍约束）/ Event-driven（事件触发）/ Unwanted（异常）/ State-driven（状态）/ Optional（可选）

---

## 1. 背景

V1.0 已上线：个性化行程生成（FR-1，含 RAG 增强与降级）、游客 Wiki 评价发布/展示/审核（FR-2~FR-5）、账号管理（FR-6）、知识库同步（FR-7）。

进入下一阶段，用户与 PO 已确认两个方向：
1. **RAG 增强**：当前 RAG 能力已具备，但"未配置密钥时的体验"与"生成结果的可解释性"不足，需补强。
2. **社区运营增强**：评价已能发布/审核，但缺乏**互动（有用投票）**与**治理（举报 + 审核工作台）**能力，运营难以判断内容质量与处理风险内容。

> 说明：本 PRD 的功能清单为**建议范围**，遵循 EARS 撰写；最终范围以评审结论为准。第 9 节列出待确认假设。

## 2. 目标与成功指标

### 2.1 业务目标
- 提升行程生成的**使用深度与信任**：让 RAG 增强"可用、可感知、可追溯"。
- 提升社区**内容质量与治理效率**：用互动信号辅助排序、用举报+审核闭环处理风险内容。

### 2.2 成功指标（与第 7 节埋点一一对应）
| 指标 | 定义 / 口径 | 目标（建议基线，待确认） |
|---|---|---|
| **北极星：周行程生成成功用户数** | 7 日内成功生成 ≥1 次行程的去重用户数 | 上线 4 周后环比 +X%（基线待埋点积累后定） |
| 行程生成成功率 | `guide_generate.success=true` 占比 | ≥ 99% |
| RAG 使用率 | `use_rag=true` 且 `source=rag` 的占比 | 配置可用后 ≥ 30% |
| RAG 降级率 | `use_rag=true` 但实际 `source=rule` 的占比 | ≤ 5% |
| 知识库来源点击率 | `guide_source_click` / `guide_generate(source=rag)` | 观察型，越高代表可解释性被认可 |
| 评价发布量 | `review_publish` 周计数 | 监控型 |
| 评价通过率 | 审核通过 / 提交总数 | 监控型（异常偏低需预警） |
| 评价互动率 | `review_helpful` / 已通过评价曝光 | 观察型 |
| 举报处理时效 | `report_resolve` 距 `review_report` 的中位时长 | ≤ 24h |

## 4. 功能清单（EARS）

### FR-9 RAG 配置引导与缺省体验（P1）

| 项目 | 描述 |
|---|---|
| 用户故事 | US-1 |
| 规则（EARS） | **Ubiquitous**：系统应始终在「定制行程」页展示当前 RAG 可用状态（可用 / 未配置）。<br>**Event-driven**：当 `RAG_API_KEY` 未配置或索引缺失时，系统应将「AI 增强生成」选项置灰并展示配置指引入口。<br>**Optional**：Where 已配置密钥且 `rag-index.json` 存在，用户勾选「AI 增强生成」后系统应走 RAG 生成（沿用 FR-1 契约）。<br>**Unwanted**：若用户勾选后 RAG 实际生成失败，则系统应降级为规则生成并提示"已切换为常规生成"，不返回 500。 |

### FR-10 行程生成溯源展示（P2）

| 项目 | 描述 |
|---|---|
| 用户故事 | US-2 |
| 规则（EARS） | **Event-driven**：当行程由 RAG 生成（`guide.rag=true`）时，系统应在行程中展示命中的知识库来源（`sources` 已有字段），支持点击展开片段。<br>**State-driven**：在「溯源」展开态下，系统应展示来源标题、所属页面与关联度（如有）。<br>**Optional**：Where 为规则生成，系统应展示"基于本地 data 与知识库的规则推荐"说明（非逐条溯源）。 |
| 依赖 | FR-1、FR-9 |

### FR-11 评价「有用」互动投票（P1）

| 项目 | 描述 |
|---|---|
| 用户故事 | US-3 |
| 规则（EARS） | **Event-driven**：当已登录用户点击某条已通过评价的「有用」时，系统应记录该用户对该评价的有用投票（同用户不可重复计票）。<br>**State-driven**：在公开评价列表中，系统应展示每条评价的有用计数，并作为排序信号之一参与倒序之外的加权。<br>**Optional**：Where 用户再次点击，系统应取消该投票并扣减计数。 |

| 依赖 | FR-3（公开列表）、FR-6 |
| 状态 | 🔲 规划中 |

### FR-12 评价举报（P1）

| 项目 | 描述 |
|---|---|
| 用户故事 | US-4 |
| 规则（EARS） | **Event-driven**：当已登录用户提交对某条已通过评价的举报（含原因）时，系统应创建一条待处理举报工单并进入审核队列。<br>**Unwanted**：若举报原因为空，或同一用户对同一评价在短时间（建议 10 分钟）内重复举报，则系统应拒绝并记录，不创建工单。<br>**Ubiquitous**：系统应确保举报行为不向被举报者泄露举报人身份。 |
| 权限 | 仅登录用户可举报；管理员不可被举报（或举报仍进队列但单独标记） |
| 依赖 | FR-3、FR-4（审核） |
| 状态 | 🔲 规划中 |

### FR-13 审核工作台增强（P2）

| 项目 | 描述 |
|---|---|
| 用户故事 | US-5 |
| 规则（EARS） | **Event-driven**：当管理员在审核后台处理一条举报工单时，系统应支持「维持评价（驳回举报）」或「下架评价（采纳举报→评价转 rejected）」两种决定，并记录操作人与时间。<br>**Event-driven**：当管理员在评价列表中勾选多条时，系统应支持批量通过 / 批量拒绝。<br>**Ubiquitous**：系统应在评价进入待审核 / 待处理举报时，对新评价的 `body` / `title` 做敏感词预检并标记高危项（词库来源待确认，见第 9 节）。<br>**Unwanted**：若批量操作中单条失败，则系统应跳过该条并汇总失败清单，不因一条失败中断整批。 |
| 权限 | 仅 admin（401/403 分级，沿用 FR-4） |
| 依赖 | FR-4、FR-12 |

## 5. 流程说明

### 5.1 RAG 增强主流程
```
游客打开「定制行程」
  ├─ 读取 meta.ragEnabled
  │    ├─ true  → 「AI 增强生成」可选可勾选
  │    └─ false → 选项置灰 + 展示配置指引入口（rag_config_view 埋点）
  └─ 提交生成
       ├─ useRag=true & 可用 → RAG 生成（guide.rag=true, sources 填充）
       └─ 失败/降级 → 规则生成 + 提示（guide.rag=false）
```

### 5.2 社区治理流程（举报闭环）
```
游客举报已通过评价（review_report）
  → 创建举报工单（pending）
  → 管理员在审核工作台查看待处理举报（敏感词预检标记）
       ├─ 驳回举报 → 评价维持 approved
       └─ 采纳举报 → 评价转 rejected（从公开列表消失）+ 记录
```

## 7. 数据指标 & 埋点需求（主笔：产品助手）

> 目标：用埋点量化第 2 节全部成功指标，为 V1.1 上线复盘提供数据。本产品零依赖（NFR-1），埋点上报须用 **Node 内置 `fetch` POST 到上报端点**，不引入 npm 包；上报端点 / 是否接分析平台待技术负责人确认（见第 9 节）。

### 7.1 埋点事件清单
| 事件名 | 触发时机 | 上报字段（建议） | 来源 |
|---|---|---|---|
| `guide_generate` | 提交 `/api/guide` 返回时 | `purpose_id`, `interests[]`, `date_range_days`, `use_rag`(bool), `rag_available`(bool), `source`(rule\|rag), `success`(bool), `fail_reason`, `guide_days`, `latency_ms` | 前端 + 服务端双写 |
| `guide_source_click` | 点击行程中的知识库来源链接 | `source_id`, `section` | 前端 |
| `rag_config_view` | 点击「AI 增强」置灰态的配置指引入口 | `entry` | 前端 |
| `review_publish` | 评价发布成功（201） | `target_type`, `rating`, `has_tags`(bool), `is_anonymous`(bool), `char_count` | 服务端 |
| `review_helpful` | 点击「有用」 | `review_id`, `target_type`, `action`(add\|remove) | 前端 + 服务端校验 |
| `review_report` | 提交举报 | `review_id`, `reason`, `reporter_role` | 服务端 |
| `review_moderate` | 审核操作 | `action`(approve\|reject), `has_note`(bool), `is_report_driven`(bool), `latency_min` | 服务端 |
| `report_resolve` | 举报处理决定 | `decision`(uphold\|dismiss) | 服务端 |
| `page_view` | 视图切换（guide/wiki/my-reviews/admin） | `view`, `user_role`(visitor\|admin\|null) | 前端 |

### 7.2 数据口径
- **去重**：同一会话内相同事件做前端防抖（如连续点击「有用」仅计首次状态变更）；服务端以"状态是否实际变化"判重。
- **用户关联**：登录用户上报带 `user_id`（哈希或 UUID，非明文）；匿名 / 游客不关联用户标识。
- **双写原则**：审核、举报、发布等**关键动作以服务端上报为准**（防前端丢失 / 篡改）；纯交互浏览（page_view、source_click）前端上报即可。
- **隐私红线**：不采集密码明文、不采集评价正文全文，仅上报计数、标签、原因分类与长度。
- **失败兜底**：上报请求失败不影响主流程（fire-and-forget + 可选本地缓冲）。
- **采样**：V1.1 全量上报即可，后续量大再评估采样。

### 7.3 看板建议（供上线复盘）
- 北极星趋势（周生成成功用户数）、RAG 使用率 vs 降级率、评价发布/通过/互动/举报时效四张卡。
