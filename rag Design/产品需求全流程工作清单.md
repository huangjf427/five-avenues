# 产品需求全流程工作清单(SOP)

> **用途**:本清单覆盖产品团队从「需求规划 → PRD 撰写 → 设计/研发评审 → 研发跟进 → 测试验收 → 上线复盘」的完整周期,作为各阶段的标准动作与交付物参考。
> **使用方式**:把本文档当作可勾选的 Checklisk,每进入一个阶段即对照该阶段的「关键动作」逐项推进;阶段结束时按「流转提醒」把成果交给下一环节。
> **通用原则**:信息不足时先追问关键缺口,不要直接编造结论;涉及排期、资源、上线范围等决策,请找对应负责人确认。

---

## 一、阶段总览

| 阶段 | 核心目标 | 关键交付物 | 主要协同角色 |
| --- | --- | --- | --- |
| ① 需求规划 | 厘清做不做得、值不值得做 | 需求概要、需求池、优先级建议、待确认问题清单 | 产品、业务、技术 |
| ② PRD 撰写 | 把需求写清楚、写无歧义 | PRD 文档、评审材料 | 产品、技术、设计、测试 |
| ③ 设计/研发评审 | 对齐方案、暴露风险与依赖 | 评审结论、待确认问题清单、设计验收点 | 产品、设计、研发、测试 |
| ④ 研发跟进 | 跟踪落地、识别延期与阻塞 | 任务拆解、进度摘要、风险清单 | 产品、研发 |
| ⑤ 测试验收 | 验证质量、判断能否上线 | 测试用例、Bug 清单、验收清单、上线风险清单 | 产品、测试、研发 |
| ⑥ 上线复盘 | 沉淀经验、规划下一版 | 复盘报告、数据观察清单、后续行动项 | 产品、研发、测试、设计、业务 |

---

## 二、阶段一:需求规划

**目标**:在动手前回答「为什么做、为谁做、值不值得做」。

### 关键动作
- [ ] 明确**目标用户**画像与人群分层(核心用户 / 边缘用户)
- [ ] 梳理**用户痛点**与典型**使用场景**(现有方案哪里不够好)
- [ ] 对齐**业务目标**与可量化的成功指标(如转化、留存、效率)
- [ ] 做**竞品调研 / 行业分析 / 现有方案对比**,找到差异化切入点
- [ ] 评估需求**价值、影响范围、实现复杂度**
- [ ] 整理**需求概要**,沉淀为**需求池**(条目化)
- [ ] 给出**优先级建议**(P0/P1/P2)与排期参考
- [ ] 列出**待确认问题清单**,找业务/技术负责人对齐关键缺口

### 核心交付物
- 需求概要
- 需求池整理表
- 优先级建议
- 待确认问题清单

### 流转提醒
- 将确定要推进的需求**创建为事项**,分配给对应负责人
- **邀请相关角色关注**,确保关键人员及时收到通知
- 重要背景资料**上传项目资料库**,形成统一上下文

---

## 三、阶段二:PRD 撰写

**目标**:用 EARS 原则把每条需求写清楚、写无歧义。

### 关键动作
- [ ] 按 **EARS 原则**撰写需求描述,确保无歧义:
  - [ ] **Ubiquitous(普遍约束)**:系统始终满足的约束 —— 「系统应……」
  - [ ] **Event-driven(事件驱动)**:事件触发的行为 —— 「当[事件]发生时,系统应……」
  - [ ] **Unwanted(异常/故障)**:异常情形处理 —— 「若[异常条件],则系统应……」
  - [ ] **State-driven(状态驱动)**:特定状态下的行为 —— 「在[状态]下,系统应……」
  - [ ] **Optional(可选)**:可选功能 —— 「在[功能]下,系统应……」
- [ ] 编写**背景、目标、用户故事**
- [ ] 输出**功能清单**与**流程说明**(主干流程 + 分支流程)
- [ ] 补充**交互说明**(页面、状态、边界)
- [ ] 明确**数据指标**与**埋点需求**(口径、触发时机、上报字段)
- [ ] 补充**边界场景、异常状态、权限规则、数据口径**
- [ ] 编写**验收标准**(可测试、可判定,避免模糊词)
- [ ] 生成 **Word 文档 / 评审材料**供团队讨论

### 核心交付物
- PRD 文档(背景 / 目标 / 用户故事 / 功能清单 / 流程 / 交互 / 数据指标 / 验收标准)
- 评审材料

### 流转提醒
- 完成后将 PRD **上传到项目资料库**
- **创建评审事项**,分配给技术负责人、设计师、测试同学
- 在事项中**附上 PRD 文档**作为交付物

---

## 四、阶段三:设计 / 研发评审

**目标**:对齐方案,提前暴露依赖、风险与待确认问题。

### 关键动作
- [ ] 整理**评审议题、待确认问题、关键依赖、风险点**
- [ ] **(产品)** 解释需求背景、业务规则、用户路径、验收标准
- [ ] **(设计)** 拆解**页面清单、状态清单、交互边界、设计验收点**
- [ ] **(研发)** 识别**技术依赖、接口问题、数据结构、兼容性、实现风险**
- [ ] 记录**评审结论**与遗留问题
- [ ] 将待办问题**拆成子项并指定负责人 / 截止时间**

### 核心交付物
- 评审结论文档
- 待确认问题清单
- 设计验收点清单

### 流转提醒
- 将评审结论**沉淀到项目资料库**
- 待办问题**拆成事项**分配给对应负责人
- 重大范围 / 技术方案变更需**找负责人确认**后再推进

---

## 五、阶段四:研发跟进

**目标**:跟踪落地进度,及时识别延期与阻塞。

### 关键动作
- [ ] 将需求**拆解为研发任务、子任务、依赖项**
- [ ] 在 **TAPD / CNB / GitHub / 工蜂** 跟踪需求 / 任务 / 代码状态
- [ ] **评审技术方案**,识别范围变更、延期风险、阻塞事项
- [ ] 整理**每日 / 每周研发进度摘要**与风险清单
- [ ] 发现阻塞**及时升级**,协调资源或调整排期

### 核心交付物
- 研发任务拆解
- 进度摘要(日 / 周)
- 风险清单

### 流转提醒
- 技术任务**拆成子事项**分配给开发同学
- 在事项中**附上 PRD / 设计稿 / 评审结论**作为依据
- 排期 / 资源决策**找对应负责人确认**

---

## 六、阶段五:测试验收

**目标**:基于 PRD 验证质量,判断是否具备上线条件。

### 关键动作
- [ ] 基于 PRD 生成**测试用例、边界场景、异常场景、回归测试范围**
- [ ] 执行测试并记录缺陷
- [ ] 撰写**规范 Bug 描述**:复现步骤 / 环境信息 / 实际结果 / 期望结果 / 影响范围
- [ ] 判断问题**是否阻断上线、是否影响核心流程、是否需调整需求范围**
- [ ] 输出**验收清单、上线风险清单、版本质量总结**

### 核心交付物
- 测试用例
- Bug 清单
- 验收清单
- 上线风险清单
- 版本质量总结

### 流转提醒
- 发现的 Bug **创建事项**分配给开发同学
- Bug 事项**关联回原需求或 PRD 文档**
- 上线范围 / 阻断判定**找产品 + 技术负责人确认**

---

## 七、阶段六:上线复盘

**目标**:沉淀经验,规划下一版本。

### 关键动作
- [ ] 总结**需求目标达成情况、上线效果、遗留问题、后续优化点**
- [ ] 整理**研发 / 测试 / 设计 / 业务侧反馈**
- [ ] 提炼**可复用经验、流程问题、下一版本需求候选项**
- [ ] 输出**复盘报告、数据观察清单、后续行动项**

### 核心交付物
- 上线复盘报告
- 数据观察清单
- 后续行动项

### 流转提醒
- 复盘结论**上传到项目资料库**
- 后续优化点**创建为新事项**,进入下一轮需求池
- 流程问题**同步给团队**,避免重复踩坑

---

## 八、贯穿全流程的通用提醒

- **阶段流转**:每个阶段完成后,主动把成果流转给下一环节角色,避免信息断点
- **附交付物**:在事项中附上对应文档(PRD / 设计稿 / 测试用例 / 评审结论 / 复盘报告)
- **加关注人**:添加关注人,让关键角色及时收到通知
- **统一上下文**:重要文档上传项目资料库,形成项目统一上下文
- **不编造**:信息不足时先追问关键缺口,不直接下结论
- **找对人**:涉及排期、资源、上线范围的决策,找对应负责人确认
